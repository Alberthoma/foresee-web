1
Debounce de renderAll() — scheduleRenderAll() Perf

Problema: 6 listeners onSnapshot (prefRef, txCol, recurCol, cardsCol, projCol, gComunRef) llamaban renderAll() directamente. Al iniciar sesión, los 6 disparaban casi simultáneamente → hasta 6 renders completos en cascada en ~50ms.

Fix: scheduleRenderAll() con debounce de 50ms. Los 6 listeners lo usan. Las acciones del usuario (switchTab, pull-to-refresh) usan renderAll() directo para respuesta inmediata.

let _renderAllTimer = null;
function scheduleRenderAll() {
  if (_renderAllTimer) clearTimeout(_renderAllTimer);
  _renderAllTimer = setTimeout(renderAll, 50);
}

commitInChunks() — límite 500 ops Firestore Arquitectura

initiateNewMonth(), resetPeriod() y cfgImportJson() usaban un único writeBatch sin límite. Firestore rechaza batches con más de 500 operaciones.

async function commitInChunks(ops) {
  const CHUNK = 499;
  for (let i = 0; i < ops.length; i += CHUNK) {
    const b = writeBatch(db);
    ops.slice(i, i + CHUNK).forEach((fn) => fn(b));
    await b.commit();
  }
}

Timezone fix en fechas Bug

Bug fix crítico — tabs bloqueados por pointer-events

📅 2026-05-25 📄 New 0017 - copia.html → New 0018.html 🔧 1 bug fix + renombre
Bug Crítico

Conversación

Los botones tab "Tarjetas de Crédito", "Saldos" y "Reportes" no responden a clics. Los otros tabs sí funcionan. ¿Cuál es la causa?
La función _syncModalState aplicaba el.style.pointerEvents = 'none' a #app-header, #tab-bar y #main-wrapper al abrir modales. Si un modal se cerraba por una ruta no estándar (guardClose en el botón X del transfer-modal), el pointer-events quedaba sin restaurarse. Fix: eliminar las líneas de pointer-events de _syncModalState — los modals position:fixed ya bloquean clics solos.
1
_syncModalState — pointer-events eliminado Bug Crítico

Síntoma: Tabs "Tarjetas de Crédito", "Saldos" y "Reportes" no respondían. El bug era intermitente — dependía de si un modal se había abierto y cerrado en la sesión actual.

Causa raíz: Una reforma anterior había añadido el.style.pointerEvents = 'none' a _syncModalState. Cuando cualquier modal se abría, se aplicaba pointer-events: none a los elementos padre del tab-bar. Si el modal cerraba por una ruta no estándar, pointer-events quedaba en none para siempre.

Por qué solo algunos tabs: La propiedad inline el.style.pointerEvents = 'none' en el elemento padre bloquea la propagación de eventos al listener delegado. El efecto era inconsistente según la historia de navegación de la sesión.

Fix: Eliminadas las líneas el.style.pointerEvents = 'none' y el.style.pointerEvents = '' de _syncModalState. Los modals tienen position: fixed; inset: 0; z-index: alto que bloquea clics visualmente. El aria-hidden y body.overflow se conservan.

checkCreditCardNotifications() — nueva función Feature

Notifica tarjetas >80% del límite, una vez por día por tarjeta.

function checkCreditCardNotifications() {
  const todayKey = new Date().toISOString().split('T')[0];
  appState.creditCards.forEach((card) => {
    const deuda = parseFloat(card.amount) || 0;
    const limite = parseFloat(card.limit) || 0;
    if (!limite || deuda / limite < 0.8) return;
    const notifKey = `notif-cc-${card.id}-${todayKey}`;
    if (localStorage.getItem(notifKey)) return;
    const pct = ((deuda / limite) * 100).toFixed(0);
    sendBrowserNotification('Foresee — Tarjeta de crédito',
      `Tarjeta ${card.bank}: ${pct}% del límite usado.`);
    localStorage.setItem(notifKey, '1');
  });
}

Bug fix — Login bloqueaba la página

Preguntas y respuestas

Ocurre algo muy extraño cuando me logeo mal y luego entro a la app. Se bloquean algunas funciones, los formularios no responden, se bloquea la página, no funcionan las teclas ESC, F12, F11 y los tab-btn de tarjeta, saldo y reportes no funcionan. ¿Por qué ocurre? Una pregunta adicional: si coloco la contraseña errada varias veces, ¿el usuario se bloquea y de ser así por cuánto tiempo?
Causa raíz 1 (principal): requestFullscreen() se llamaba dentro de handleLogin() inmediatamente después de que Firebase autenticara, mientras auth-modal todavía estaba visible. El navegador entraba en fullscreen con el modal activo. En fullscreen: ESC sale del fullscreen (no cierra modal) → el usuario percibe "ESC no funciona". F11 intenta alternar fullscreen del navegador → conflicto con API. Causa raíz 2: sin guard de doble envío — Enter en password disparaba el form aunque el botón estuviera deshabilitado, dejando el estado UI inconsistente. Sobre Firebase: sí bloquea con auth/too-many-requests tras ~10 intentos fallidos. Backoff exponencial, típicamente unos minutos.

handleLogin() / handleRegister() — reescritos con guards Bug Crítico
  • requestFullscreen() eliminado de handleLogin()
  • Flag let _authInProgress = false — bloquea dobles envíos en login y registro
  • finally block en ambas funciones: setAuthLoading(false) siempre ejecuta
  • Mensaje específico para auth/too-many-requests: "Demasiados intentos fallidos. Espera unos minutos e intenta de nuevo."
let _authInProgress = false;

async function handleLogin(email, password) {
  if (_authInProgress) return;
  clearAuthErrors();
  _authInProgress = true;
  setAuthLoading(true);
  try {
    await signInWithEmailAndPassword(auth, email, password);
  } catch (err) {
    const msg = err.code === 'auth/too-many-requests'
      ? 'Demasiados intentos fallidos. Espera unos minutos e intenta de nuevo.'
      : 'Correo o contraseña incorrectos.';
    document.getElementById('auth-password-error').textContent = msg;
  } finally {
    _authInProgress = false;
    setAuthLoading(false);
  }
}

Fullscreen post-login restaurado

Preguntas y respuestas

Por ahora con la modificación está funcionando bien, pero antes desde que hice la aplicación la pantalla abría en fullscreen al iniciar la app y quiero que la aplicación después de loguearse abra fullscreen. Analiza y dame una respuesta.
El problema de fondo es que requestFullscreen() requiere transient user activation — una ventana de ~1 segundo. Los callbacks de Firebase (onAuthStateChanged, onSnapshot) son macrotasks asíncronas que llegan fuera de esa ventana. No se puede llamar requestFullscreen() directamente ahí de forma confiable. La solución correcta: ya existe #fullscreen-prompt en el HTML (nunca se había activado). Es una pantalla de bienvenida "Iniciar Aplicación". El clic del usuario sí es un user gesture → requestFullscreen() funciona de forma confiable. Flujo: Login → datos cargan → pantalla bienvenida → usuario hace clic → fullscreen confiable.
